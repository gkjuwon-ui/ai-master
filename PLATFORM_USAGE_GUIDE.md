# AI Master Platform 사용 가이드 (구독/가성비/운영 팁)

이 문서는 코드베이스에 정의된 **구독 티어(STARTER/PRO/APEX)** 및 seed에 포함된 **에이전트 가격/티어/도메인 구조**를 기준으로,
"가장 효율적인 구독 플랜", "가장 가성비 좋은 에이전트 조합", "플랫폼을 잘 쓰는 방법"을 실전 위주로 정리합니다.

---

## 1) 구독 플랜: 무엇이 가장 효율적인가?

백엔드 기준( `backend/src/services/subscriptionService.ts` ) 구독 티어는 아래 3가지입니다.

- **Starter**
  - 가격: `$9.99/mo` (연간: `$7.99/mo`)
  - 실행 한도: `월 50회`
  - 실행 시간: 5분(설명 기준)
  - 특징: "플랫폼을 본격적으로 써보고 싶은데 과금 리스크는 낮추고 싶다"에 최적.

- **Pro (Best Value / 목표 상품)**
  - 가격: `$29.99/mo` (연간: `$23.99/mo`)
  - 실행 한도: `월 300회`
  - 실행 시간: 10분(설명 기준)
  - 특징: 대부분의 유저가 체감하는 "가성비 최적점". 실행 횟수/시간이 실사용 임계치를 넘기기 때문에 작업이 끊기지 않음.

- **Apex (Anchor/Decoy 성격 포함)**
  - 가격: `$99.99/mo` (연간: `$79.99/mo`)
  - 실행 한도: `무제한`
  - 실행 시간: 무제한(설명 기준)
  - 특징: "진짜 하루 종일 돌릴" 운영/업무 자동화 또는 팀 단위 사용에 의미가 큼.

### 추천 결론

- **대부분의 개인/소규모 팀 최적**: **Pro(연간)**
  - 월 실행 300회는 "테스트/실행/재시도"까지 포함해도 꽤 여유롭고, 시간 10분은 복잡한 작업에서 체감 차이가 큼.

- **가볍게 써보며 습관 들이기**: **Starter(연간)**
  - 월 50회는 실험/학습에는 충분. "내 작업이 플랫폼에 맞는지" 검증할 때 좋음.

- **무제한이 실질적으로 필요한 경우만 Apex**
  - 운영 자동화(모니터링/정기 작업), 반복적인 이슈 트리아지/리포트 작성, 다량의 실행이 필요한 QA/리그레션 체크 등에 적합.

## 2) "가성비 에이전트" 추천 (가격/티어 관점)

### 가성비 기준
- **성공률이 낮은 에이전트는 싸도 비싸다**
  - 반복 실행/재시도/무의미 행동이 누적되면 실행 한도만 깎이고 결과 품질은 낮아짐.
- 따라서 "가격"보다 **작업 타입에 맞는 도메인/티어 선택**이 가성비를 결정.

### Seed 기준(예시) 빠른 추천
(가격은 seed에 포함된 샘플 데이터 기준)

#### 1) 일상 자동화/잡무 처리 (가성비)
- **FREE: ClickBot (clickbot)**
  - 클릭/간단 자동화 실험용
- **$7.99: Scrappy (scrappy)**
  - 저가형이지만 반복 작업/단순 작업에 투입하기 좋음
- **$12.99~$24.99: Taskmaster/Deployer 계열**
  - "단계가 있는" 작업(설치/배포/정리)은 이 구간이 체감 가성비가 좋음

#### 2) 코딩 (비용 대비 효율)
- **$9.99: Codewatch (codewatch)**
  - 저렴한 코딩 작업 보조 (리뷰/간단 수정/에러 확인 루프에 투입)
- **$39.99: Architect (architect)**
  - 프로젝트 구조 설계/리팩토링 설계 같은 "한 번 잘하면 큰 비용 절감" 영역에 효율적
- **상위(비싸지만 확실)**: Apex Coder ($129.99)
  - 복잡한 다파일 변경, 테스트/실행/수정 루프가 많은 경우에는 오히려 총 비용이 줄 수 있음(실패율 감소)

#### 3) 리서치/문서 작성
- **FREE: Quill (quill)** / **$14.99: Scribe (scribe)**
  - 문서 생성/정리류 가성비가 가장 좋음
- **Apex Researcher ($79.99)**
  - 고난도 조사/긴 보고서/근거 추적이 필요할 때만

#### 4) 데이터 분석
- **$34.99: DataForge (dataforge)**
  - EDA/리포트/간단 모델링의 현실적인 가성비 구간
- **Apex Analyst ($99.99)**
  - 큰 데이터/정교한 통계/대시보드 자동화 등 "사람 시간"을 크게 줄이는 케이스에 적합

### 요약: "한 번에 성공해야 하는 작업"일수록 상위 티어가 가성비
- 단순/반복/실험: 저가/무료로 충분
- 실패 비용이 큰 작업(배포/대규모 수정/리서치 보고서): 상위 티어가 결과적으로 저렴

## 3) 플랫폼을 잘 쓰는 방법 (성능이 잘 나오는 운영 팁)

### A. 프롬프트는 "목표 + 완료 조건 + 금지"가 핵심
아래 3가지를 항상 포함하면 성공률이 크게 올라갑니다.

- **목표**: 무엇을 만들/얻을지
- **완료 조건(Verification)**: "어떻게 되면 끝"인지
- **금지/주의**: 건드리면 안 되는 것

예시:
- 목표: "로그인 페이지 버그 수정"
- 완료 조건: "테스트 통과 + 브라우저에서 로그인 성공"
- 금지: "DB 마이그레이션 금지"

### B. GUI 작업은 "관찰→행동" 루프가 중요
- 화면 요소 인식 실패 시 **무의미 클릭**이 폭발하기 쉬움
- 이 코드베이스는 SoM(요소 오버레이) + semantic fallback(VLM 요소 추출)을 넣어 해결했지만,
  그래도 안정성을 위해:
  - **작업 창을 최대화**
  - **불필요한 창 닫기**
  - **스크롤/페이지 이동을 최소화**
  - **한 번에 한 앱** 원칙을 지키면 성능이 좋아짐

### C. 실행 한도(Executions)를 아끼는 전략
- "큰 목표"를 한 번에 던지기보다는 **2~3단계로 나눠서**
  - 1단계: 탐색/상태 파악
  - 2단계: 작업 수행
  - 3단계: 검증/정리
- 실패했을 때 재시도를 줄이는 구조라 실행 한도 대비 효율이 훨씬 좋음

### D. 추천 워크플로우 (작업 타입별)

#### 코딩 작업
- 1) 에러 재현 조건 명시
- 2) 수정 범위(파일/모듈) 제한
- 3) 테스트 명령(또는 빌드 명령) 제공
- 4) 성공 기준(테스트 통과, 빌드 성공) 명시

#### 리서치/문서
- 1) 산출물 포맷(MD/슬라이드/요약/표) 지정
- 2) 근거 요구(출처/링크/인용) 명시
- 3) 길이/섹션 구조 지정

#### 운영 자동화
- 1) 반복 주기(매일/매주/이벤트 발생 시)
- 2) 실패 시 동작(알림/재시도/중단)
- 3) 로그/결과 저장 위치

## 4) 티어 선택 빠른 규칙

- **F / 무료**: 단순 반복, 기능 탐색, UI 클릭봇 수준
- **C~B**: 일반 업무(문서/정리/간단 자동화)
- **A~S**: 실패 비용 큰 작업(배포/대규모 변경/중요 보고서)
- **S+**: "무조건 성공"에 가까운 퀄리티가 필요한 경우에만

## 5) 추천 조합 (구독 × 에이전트)

- **Starter(연간) + 저가 에이전트 1~2개**
  - "내 업무가 에이전트에 맞는지" 검증 및 습관화에 최적

- **Pro(연간) + 중가 에이전트(Architect/DataForge 등) + 필요 시 S+ 스팟 사용**
  - 실사용 가성비 최적점

- **Apex + 자동화/운영 중심 에이전트**
  - 무제한 실행이 실제로 비용 절감으로 이어지는 케이스에만

## 6) FAQ

### Q. "구독하면 모든 에이전트가 바로 공짜로 열리나요?"
- 코드 주석 기준, 구독은 "직접 접근 권한"을 자동으로 주기보단 **$0로 claim할 자격**을 주는 구조입니다.
  - 즉, 구독 후에도 각 에이전트를 구매/claim 플로우로 가져오는 UX가 전제됩니다.

### Q. 실행 한도가 부족해요
- **Pro**가 사실상 대부분 케이스의 해결책(300회)이고,
- 정말로 실행 수/시간이 업무량에 비례해 늘어나는 팀이면 **Apex**가 낫습니다.

## 부록) 개발/운영자 체크리스트 (성능/품질)

- 비전(vision) 미지원 모델로 GUI 작업을 돌리지 말 것

- 실패율이 높으면 저가 에이전트 다회 실행보다 상위 티어 1회가 싸게 먹힘
- 실행 로그(METRIC 포함)를 기준으로 회귀/품질을 관리할 것
