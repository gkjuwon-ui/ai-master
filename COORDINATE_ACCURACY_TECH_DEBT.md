# 좌표 추출 & 클릭 정확도 — 기술 부채 보고서

> 작성일: 2026-02-16 (v2 — 1920×1080 환경 근본 원인 추가)  
> 심각도: **CRITICAL** — 에이전트의 화면 조작 능력을 근본적으로 제한하는 구조적 결함

---

## 요약

에이전트가 화면에서 올바른 위치를 클릭하지 못하는 원인은 **두 계층**이다:

**계층 1 — 1920×1080 @ 100% DPI에서도 발생하는 근본 결함**:
- SoM 엣지 감지가 UI 요소를 정확히 잡지 못함 (근본적 알고리즘 한계)
- 28px 그리드 양자화로 클릭 좌표가 14px 이상 빗나감
- LLM (Gemini)이 이미지 좌표를 정확히 추정하지 못함 (모델 내부 해상도 불일치)
- 좌표 검증/보정 레이어가 아예 없음
- 탐욕적 JSON 파서가 LLM 응답에서 잘못된 좌표를 추출할 수 있음
- Windows UI Automation API 미사용 (100% 정확한 요소 위치를 제공하는 API가 존재함에도)

**계층 2 — 비-1920×1080 환경에서 추가 발생하는 DPI/해상도 문제**:
- SoM 좌표공간 ≠ pyautogui 좌표공간
- DPI 스케일 계산하고도 미사용
- LLM 이미지 좌표 → 화면 좌표 변환 부재

## 1920×1080에서도 클릭이 엉뚱한 곳에 가는 이유

### 전체 파이프라인

```
스크린샷 캡처 (mss, 1920×1080) 
    → SoM 엣지 감지: 가우시안 블러 → 엣지 필터 → 28px 그리드 → BFS 클러스터링
    → 바운딩 박스 생성: 클러스터의 min/max 그리드 셀 × 28px
    → 📛 문제1: 바운딩 박스가 실제 UI 요소와 안 맞음
    → 중심 좌표 계산: cx = x + w/2, cy = y + h/2
    → 📛 문제2: 중심점이 클릭 가능한 영역이 아님
    → 어노테이션 이미지: 번호 뱃지 + 색상 테두리 그려서 LLM에 전송
    → 📛 문제3: 뱃지가 UI를 가려서 LLM이 잘못 인식
    → LLM 응답: "click_element {id: 5}" 또는 "click {x: 960, y: 540}"
    → 📛 문제4: LLM이 5번이라고 생각하는 것이 실제 5번이 아닐 수 있음
    → 📛 문제5: 원시 좌표를 줄 때 LLM 내부 해상도와 실제 해상도가 다름
    → _parse_actions(): 응답에서 액션/좌표 추출
    → 📛 문제6: 탐욕적 JSON 파서가 OBSERVATION 섹션의 좌표를 잡을 수 있음
    → pyautogui.click(x, y): 추출된 좌표 그대로 클릭
    → 📛 문제7: 좌표 검증/보정 로직 없음
```

## 근본 원인 #1 — SoM 엣지 감지의 알고리즘 한계 (CRITICAL)

**파일**: [som_engine.py](agent-runtime/core/som_engine.py)

현재 알고리즘:
```
이미지 → 그레이스케일 → 가우시안 블러(r=1) → FIND_EDGES → 28px 그리드 → BFS
```

이 알고리즘은 **"엣지 밀도가 높은 영역"**을 찾는 것이지, **"클릭 가능한 UI 요소"**를 찾는 것이 아니다.

#### 실패 사례들:

| 상황 | 결과 |
|------|------|
| 텍스트 링크 (파란색 텍스트, 엣지 약함) | ❌ 감지 실패 — 텍스트는 엣지가 약함 |
| 큰 이미지 배너 (엣지 많음) | ❌ 거대한 "region"으로 감지 — 중심점이 무의미 |
| 플랫 디자인 버튼 (단색, 엣지 없음) | ❌ 감지 실패 — 엣지가 없으므로 보이지 않음 |
| 텍스트 단락 (글자마다 엣지) | ❌ 하나의 큰 버튼/리전으로 감지 — 실제로는 클릭 불가 |
| 드롭다운 메뉴 항목 | ❌ 배경과 구분이 약하면 감지 실패 |
| 인접한 작은 아이콘들 | ❌ 하나의 큰 리전으로 합쳐짐 |
| Chrome 탭 / 주소창 | ⚠️ 간혹 감지되나, 바운딩 박스가 여러 탭을 아우름 |

**핵심 문제**: 웹 페이지의 "검색 결과 3번째 링크를 클릭해라"와 같은 지시가 오면, 그 링크 텍스트의 엣지가 약해서 SoM이 감지하지 못하거나, 인접 요소와 합쳐져서 엉뚱한 중심점을 잡는다.

## 근본 원인 #2 — 28px 그리드 양자화 오차 (CRITICAL)

**파일**: [som_engine.py](agent-runtime/core/som_engine.py) 라인 72, 233-236

```python
CELL_SIZE = 28     # 분석 그리드 셀 크기

# 바운딩 박스 생성 (라인 233-236)
ex = min_c * CELL_SIZE     # ex는 항상 28의 배수
ey = min_r * CELL_SIZE     # ey도 항상 28의 배수
ew = (max_c - min_c + 1) * CELL_SIZE
eh = (max_r - min_r + 1) * CELL_SIZE
```

모든 바운딩 박스는 28px 그리드에 스냅된다. 실제 UI 요소가 (110, 205)에 시작해도, 바운딩 박스는 (84, 196)에서 시작한다 (28의 배수).

**구체적 예시**:

실제 버튼 위치: (112, 350) ~ (230, 380), 크기 118×30px  
→ 그리드 셀: col 4~8, row 12~13  
→ 바운딩 박스: (112, 336) ~ (252, 392), 크기 140×56px  
→ 계산된 중심: (182, 364)  
→ **실제 버튼 중심: (171, 365)** — 11px 차이

이 정도는 괜찮지만, 최악의 경우:

실제 아이콘 위치: (500, 200) ~ (524, 224), 크기 24×24px  
→ 그리드 셀: col 17~18, row 7~8  
→ 바운딩 박스: (476, 196) ~ (532, 252), 크기 56×56px  
→ 계산된 중심: (504, 224)  
→ **바운딩 박스가 24×24 아이콘 대비 2.3배 커서 주변 빈 공간 중심을 클릭 가능**

28px 양자화는 작은 UI 요소에 대해 **최대 ±14px 오차 + 바운딩 박스 팽창**을 유발한다. 작은 버튼이나 아이콘은 이것만으로 클릭 실패.

## 근본 원인 #3 — LLM (Gemini) 좌표 추정 한계 (CRITICAL)

LLM이 `click_element`이 아닌 원시 좌표 `click {"x": ..., "y": ...}`를 사용할 때:

### 문제 3-A: Gemini의 내부 이미지 해상도

Gemini 2.5 Flash는 입력 이미지를 **내부적으로 리사이즈**하여 처리한다. 1920×1080 이미지를 보내도, 모델 내부에서는 다른 해상도로 처리될 수 있다. 좌표를 요청하면:

- 모델이 768×768 내부 해상도로 이미지를 처리한 경우
- "화면 중앙"을 (384, 384)로 추정
- 이를 이미지 기준으로 스케일하면 (960, 540)이 되어야 하지만
- 모델이 직접 (384, 384)를 출력할 수 있음
- → 실제 클릭: 화면의 좌측 상단 근처

### 문제 3-B: 좌표 명시 부재

시스템 프롬프트에 좌표 체계에 대한 **명확한 정의가 없다**:

```
현재 프롬프트: ACTION: click PARAMS: {"x": 500, "y": 300}
```

- "500, 300이 무엇을 의미하는지" 설명 없음
- "좌표는 반드시 0~1920 범위의 절대 픽셀 좌표여야 한다" 같은 명시 없음
- Grid 설명에 `Screen: 1920×1080px`라고 쓰여있지만, LLM이 이걸 좌표 기준으로 인식하는 보장 없음
- Gemini 모델이 자체적으로 정규화된 좌표(0~1 또는 0~1000)를 사용할 가능성

### 문제 3-C: 이미지 vs 텍스트 좌표 불일치

LLM에게 두 가지 정보가 동시에 주어진다:
1. **이미지**: 번호 뱃지가 그려진 스크린샷 (시각적 위치)
2. **텍스트**: SoM 설명 `[3] 🔘 button at ( 500, 300) 56x28 small zone=middle-center`

LLM이 이미지를 보고 [3]의 위치를 시각적으로 추정한 좌표 ≠ 텍스트 설명의 좌표. 
LLM이 둘 중 어느 것을 참조하느냐에 따라 완전히 다른 좌표가 출력된다.

## 근본 원인 #4 — 좌표 검증/보정 레이어 부재 (HIGH)

**파일**: [agent_intelligence.py](agent-runtime/core/agent_intelligence.py) `ActionValidator` 클래스

`ActionValidator.validate()`는 다음만 검사:
- 리서치 작업에서 파일 탐색기 열기 차단
- 같은 액션 3회 반복 차단
- 브라우저에서 URL 입력 시 Ctrl+L 선행 권장

**좌표 검증은 전혀 하지 않는다**:
- `x, y`가 정수인지 확인 안 함 → `{"x": 0.5, "y": 0.3}` (정규화 좌표)가 그대로 통과
- `x, y`가 화면 범위 내인지 확인 안 함 → `{"x": 5000}` 통과
- `x, y`가 `float`인지 `int`인지 확인 안 함
- `x, y`가 0~1 범위면 정규화 좌표로 해석하는 로직 없음
- SoM 요소 주변 N픽셀 이내인지 교차 검증하는 로직 없음

`_resolve_coordinates()` ([os_controller.py](agent-runtime/core/os_controller.py) 라인 496)는 화면 바운드로 **클램프만** 수행:
```python
resolved["x"] = max(0, min(int(resolved["x"]), self.screen_width - 1))
```
이것은 (5000, 3000) → (1919, 1079)로 만들어버린다. 완전히 엉뚱한 좌표를 화면 끝에 클램프시켜서 **화면 우하단 구석을 클릭**하게 됨.

## 근본 원인 #5 — 탐욕적 JSON 파서 (HIGH)

**파일**: [engine.py](agent-runtime/core/engine.py) 라인 1896-1908

```python
# 2) Inline JSON snippet (object/array)
starts = [p for p in (stripped.find("["), stripped.find("{")) if p >= 0]
if starts:
    start = min(starts)
    end = max(stripped.rfind("]"), stripped.rfind("}")) + 1
    if end > start:
        snippet = stripped[start:end]
        obj = json_module.loads(snippet)
```

이 파서는 **응답 전체에서 첫 번째 `[`/`{`부터 마지막 `]`/`}`까지**를 잡아서 JSON 파싱을 시도한다.

#### 위험한 시나리오:

LLM 응답:
```
**OBSERVATION**: I see elements [1] at (200,100), [2] at (400,300). The browser shows...
**THINKING**: I should click element [1]
**ACTION**: click_element
**PARAMS**: {"id": 1}
```

파서 동작:
1. 펜스 JSON (전략 1): 트리플 백틱 없음 → 건너뜀
2. 인라인 JSON (전략 2):
   - 첫 `[` → OBSERVATION의 `[1]` → 위치 30~40
   - 마지막 `}` → PARAMS의 `}` → 위치 끝
   - `snippet = "[1] at (200,100), [2] at (400,300)...{"id": 1}"` → JSON 파싱 실패 → 다행히 전략 3으로 넘어감

하지만 이 시나리오에서는 실패:
```
Here is my action: {"type": "click", "params": {"x": 960, "y": 540}}
But I considered alternative coordinates at [400, 300] as well.
```

파서:
1. 첫 `{` → `{"type": "click"...}` → 위치 20
2. 마지막 `]` → `[400, 300]` → 위치 끝
3. `snippet = '{"type": "click", "params": {"x": 960, "y": 540}}\nBut...[400, 300]'` → JSON 파싱 실패

대부분 실패하지만, **간헐적으로 잘못된 JSON이 우연히 파싱 성공하면, 엉뚱한 좌표가 추출된다.**

## 근본 원인 #6 — Windows UI Automation API 미사용 (HIGH)

Windows에는 **UI Automation API**가 내장되어 있다:
```python
# pywinauto 또는 comtypes로 사용 가능
import uiautomation as auto
window = auto.WindowControl(Name="Google Chrome")
buttons = window.ButtonControl(searchDepth=5)
print(buttons.BoundingRectangle)  # ← 100% 정확한 좌표
```

이 API는:
- 모든 Windows 애플리케이션의 UI 트리 접근 가능
- 버튼, 텍스트박스, 링크 등의 **정확한 바운딩 박스** 반환
- 요소의 이름, 타입, 상태(활성화/비활성화) 제공
- **엣지 감지보다 100배 정확**

현재 코드에서는 이 API를 **전혀 사용하지 않는다**. `pyautogui`만 사용하고, 요소 감지는 순전히 이미지 기반 엣지 감지에 의존.

## 근본 원인 #8 — 프롬프트 구조 문제 (MEDIUM)

**파일**: [engine.py](agent-runtime/core/engine.py) `_build_system_prompt()` 라인 1513-1820

### 문제 8-A: 시스템 프롬프트 크기

시스템 프롬프트가 **~6,500 토큰** (약 300줄). 이 안에:
- Windows 앱 가이드 (~50줄)
- 인지 프레임워크 (~50줄)
- 비주얼 결정 트리 (~80줄)
- 아이언 룰 (~15줄)
- 액션 목록 (~40줄)

이 모든 것이 **매 턴마다** 전송된다. LLM의 주의가 분산되어 좌표 정확도에 집중하지 못할 수 있다.

### 문제 8-B: 좌표 지시 부족

프롬프트에 "좌표를 어떻게 결정하라"는 구체적 지시가 없다:
- "element ID를 사용하라"와 "coordinate로 클릭하라" 두 옵션만 제시
- "좌표는 반드시 스크린샷의 절대 픽셀 좌표여야 한다" 명시 없음
- "좌표를 추정할 때 스크린샷의 가로 1920px, 세로 1080px 기준으로 하라" 명시 없음
- 좌표 추정 방법에 대한 가이드 전무

## 핵심 결론

> **1920×1080 @ 100% DPI에서도 클릭이 엉뚱한 곳에 가는 이유는 DPI/해상도 문제가 아니다.** 
>
> 근본 원인은:
> 1. **SoM 엣지 감지가 UI 요소가 아니라 "엣지 밀도가 높은 영역"을 찾는다** — 실제 버튼/링크/입력칸을 정확히 잡지 못한다
> 2. **28px 그리드 양자화로 바운딩 박스가 실제 요소보다 크고 중심점이 빗나간다**
> 3. **LLM이 원시 좌표를 출력할 때, 모델 내부 이미지 해상도와 실제 해상도 차이로 좌표가 크게 빗나간다**
> 4. **좌표 검증/보정 로직이 전혀 없어서, 잘못된 좌표가 그대로 pyautogui에 전달된다**
> 5. **Windows UI Automation API(100% 정확한 요소 좌표 제공)를 전혀 사용하지 않는다**
>
> 가장 효과적인 수정은 **Windows UI Automation API 통합**이다. 이것 하나로 SoM 감지 정확도, 좌표 양자화, LLM 좌표 추정 문제를 모두 우회할 수 있다.
